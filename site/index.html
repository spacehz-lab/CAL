Evidence-gated capability acquisition

From provider-specific actions to verified, reusable capabilities.

CAL separates provider-independent capabilities from provider-specific bindings, verifies candidate bindings with local evidence, and promotes only validated records for later reuse. The current reference system and evaluation use local CLI providers.

Acquisition

Acquire capability records from provider surfaces. Model output remains a candidate; local probes and deterministic checks decide which bindings are promoted.

Provider surface

packnote manifest, field extraction, note packaging
corp-redact email redaction, token redaction, line-preserving output
jq / tar / datapick provider-specific commands and option surfaces
>

Promote process

Observe
Read help, examples, and local behavior.
Propose
LLM proposes capability + binding candidates.
Probe
Execute candidates against controlled fixtures.
Verify
Failed checks become negative evidence.
Promote
Only verified bindings become records.
>

Capability records

manifest.create provider: packnote
pii.redact provider: corp-redact
json.field.query provider: jq or datapick

One provider can promote multiple capability records; invalid candidates stay out.

Reuse

Select promoted bindings by intent. Reuse starts from evidence-backed capability records rather than generating a provider command from scratch.

Reuse can work across providers because capability identity is separated from bindings.

Provider-first vs capability-first

Provider-first starts from available tools. Capability-first starts from verified task records and binds them to providers.

Provider-first tool use

Start from a provider surface and resolve an action for the current task.

1
Start from a provider surfaceActions are exposed through a CLI, API, tool schema, app, or similar provider-specific interface.
2
Interpret provider-specific actionsA task may still require provider documentation, syntax, or local context.
3
Use the immediate resultA successful execution does not automatically produce a verified, reusable capability record.
4
Revisit when context changesProvider drift, local versions, or unavailable functions may require re-observation.

Capability-first with CAL

Start from a verified task capability and select a provider binding at reuse time.

1
Acquire from provider surfacesObserve help and behavior, then propose candidate bindings.
2
Probe and verifyLocal execution and deterministic checks decide whether the binding works.
3
Promote capability recordsOnly evidence-backed bindings are persisted as capability records.
4
Reuse by intentcalctl use selects over verified records, not raw provider guesses.

Benchmark scope

The suite is intentionally controlled and auditable. It is meant to test capability acquisition and reuse mechanics, not to claim coverage over every possible command-line program.

Known CLIs Common tools such as JSON, archive, hashing, and text-processing commands.
Uncommon providers Eval-local enterprise-style CLIs including acmejson, corp-redact, datapick, and packnote.
Drift providers Tools that advertise plausible surfaces but return semantically wrong outputs.
Held-out reuse Reuse rounds run after acquisition using only intent and runtime inputs.

Reproduce the local checks

The public repository includes deterministic validation and replay paths. Live LLM runs require an OpenAI-compatible endpoint. 

make build
python3 evals/cli-capability/runner/validate.py
python3 -m unittest evals/cli-capability/runner/run_test.py

python3 evals/cli-capability/runner/run.py \
  --mode replay \
  --experiment acquisition,verification_failure,repeated_reuse,capability_structure \
  --level focus \
  --jobs 8 \
  --calctl build/bin/calctl \
  --cald build/bin/cald

Limitations

CAL is evaluated on a controlled CLI seed benchmark. The reported live LLM results use Kimi kimi-k2.7-code; broader claims need more models, more real third-party providers, and larger external task suites.

Controlled CLI benchmark, not a universal CLI leaderboard
CAL is a local reference system for verified capability acquisition and reuse. Source and sanitized artifacts are available at github.com/spacehz-lab/CAL.
